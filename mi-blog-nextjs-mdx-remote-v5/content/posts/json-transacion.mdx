Para una **API REST moderna**, lo más recomendable es **enviar un solo JSON que represente toda la factura** (encabezado + detalles). Esa es la práctica que más verás en APIs bien diseñadas.

## ¿Por qué un solo JSON?

Porque una factura es una **unidad de negocio**. Desde el punto de vista del negocio, no tiene sentido crear primero el encabezado y luego los detalles por separado. Lo correcto es que:

- Se guarde toda la factura.
- O no se guarde nada.

Esto se conoce como **una operación atómica**, normalmente implementada con una transacción en la base de datos.

---

## Ejemplo recomendado

Supongamos que tienes las tablas:

- Invoice
- InvoiceDetail

El endpoint sería:

```http
POST /api/invoices
```

Y recibiría un único JSON:

```json
{
  "customerId": 15,
  "invoiceDate": "2026-07-17",
  "paymentMethod": "CONTADO",
  "observations": "Factura de prueba",
  "details": [
    {
      "productId": 10,
      "quantity": 2,
      "unitPrice": 15000,
      "discount": 0
    },
    {
      "productId": 25,
      "quantity": 1,
      "unitPrice": 50000,
      "discount": 5000
    }
  ]
}
```

En .NET tendrías un DTO como:

```csharp
public class CreateInvoiceDto
{
    public int CustomerId { get; set; }

    public DateTime InvoiceDate { get; set; }

    public string PaymentMethod { get; set; }

    public string? Observations { get; set; }

    public List<CreateInvoiceDetailDto> Details { get; set; } = [];
}
```

Y el detalle:

```csharp
public class CreateInvoiceDetailDto
{
    public int ProductId { get; set; }

    public decimal Quantity { get; set; }

    public decimal UnitPrice { get; set; }

    public decimal Discount { get; set; }
}
```

# ¿Cómo se procesa?

En el servicio:

```text
Recibir JSON

↓

Validar DTO

↓

Validar cliente

↓

Validar productos

↓

Crear entidad Factura

↓

Crear entidades Detalle

↓

Iniciar transacción

↓

Guardar encabezado

↓

Guardar detalles

↓

Commit

↓

Responder 201 Created
```

Si falla cualquiera de los pasos:

```text
Rollback

↓

No queda información incompleta
```

# ¿Y dos JSON separados?

Algunas personas hacen algo así:

```http
POST /api/invoices
```

Devuelve:

```json
{
  "invoiceId": 250
}
```

Después llaman:

```http
POST /api/invoices/250/details
```

Luego:

```http
POST /api/invoices/250/details
```

y así sucesivamente.

Este diseño suele tener varios inconvenientes:

- Si falla la segunda llamada, queda un encabezado sin detalles.
- Necesitas lógica adicional para eliminar facturas incompletas.
- Genera más tráfico de red.
- Complica el frontend.

Por eso **no suele ser la mejor opción para crear una factura completa**.

---

# ¿Cuándo sí tendría sentido usar endpoints separados?

Cuando los detalles realmente se administran de forma independiente.

Por ejemplo:

```
POST   /api/invoices
POST   /api/invoices/{id}/details
PUT    /api/invoices/{id}/details/{detailId}
DELETE /api/invoices/{id}/details/{detailId}
```

Esto es útil cuando la factura ya existe y el usuario necesita modificarla antes de finalizarla.

---

# ¿Qué hacen los ERPs?

La mayoría de los ERP modernos (SAP, Microsoft Dynamics 365, Odoo, Oracle NetSuite, entre otros) utilizan un enfoque similar:

- Para **crear** una factura: envían el documento completo (encabezado + líneas).
- Para **editar** una factura existente: permiten agregar, modificar o eliminar líneas mediante operaciones específicas.

# ¿Qué te recomiendo para tu proyecto?

Como sé que estás desarrollando una aplicación de facturación con:

- .NET 8
- SQL Server
- Dapper o EF Core
- Arquitectura en capas
- React

Yo implementaría:

```text
POST /api/invoices
```

Recibiendo un único DTO:

```text
CreateInvoiceDto
    ├── Datos del encabezado
    └── List<CreateInvoiceDetailDto>
```

Y dentro del servicio:

1. Validar el DTO (por ejemplo con FluentValidation).
2. Verificar que existan cliente y productos.
3. Calcular subtotales, impuestos, descuentos y total **en el servidor**, sin confiar en los valores enviados por el cliente.
4. Abrir una transacción.
5. Guardar el encabezado.
6. Guardar todos los detalles.
7. Confirmar la transacción.
8. Devolver el identificador de la factura creada.

Este enfoque mantiene la consistencia de los datos y es el más utilizado en APIs REST empresariales modernas.

En tu caso, estás tocando tres partes del sistema (Encabezado, Cuerpo/Detalles y Stock de Producto), pero todas forman parte de una **única unidad lógica de negocio: "Registrar una Venta/Factura"**.

---

**No, en absoluto.** Si vas a trabajar con una sola tabla, implementar el patrón **Unit of Work (UoW)** de forma manual es una de las prácticas de sobreingeniería más comunes (y desaconsejadas) en .NET.

De hecho, en .NET 10 (y desde hace varias versiones de EF Core), la recomendación general es **no crear capas personalizadas de Unit of Work ni de Repositorio Genérico**, a menos que tengas una razón de peso muy específica.

Aquí te explico los motivos técnicos y qué debes hacer con las transacciones en tu caso:

---

## 1. EF Core ya es tu Unit of Work y tu Repositorio

Cuando usas Entity Framework Core:

* **`DbContext`** ya actúa como el patrón **Unit of Work**. Mantiene el seguimiento de los cambios (Change Tracker) de todas las entidades involucradas en una operación.
* **`DbSet<T>`** ya actúa como el patrón **Repository**. Tiene los métodos `Add`, `Remove`, `Update` y permite hacer consultas LINQ.

Si creas una clase llamada `UnitOfWork` que adentro solo tiene un `_context.SaveChangesAsync()`, lo único que estás haciendo es envolver una herramienta que ya funciona perfectamente dentro de otra capa innecesaria (un *wrapper* del *wrapper*), lo que añade código repetitivo (*boilerplate*) sin aportar ningún valor.

---

## 2. ¿Qué pasa con las transacciones si es una sola tabla?

Aquí hay una confusión común: **No necesitas Unit of Work para tener transacciones.** Cuando tú haces operaciones en EF Core sobre una sola tabla (o incluso sobre varias):

* **`DbContext.SaveChangesAsync()` es atómico por defecto.** Esto significa que todo lo que hayas modificado o agregado antes de llamar a ese método se ejecutará dentro de una transacción de base de datos automática. Si una fila falla, todo el bloque de esa llamada hace *rollback*.

### El único escenario donde sí necesitas configurar transacciones manualmente:

Si en tu lógica de negocio necesitas hacer **múltiples llamadas separadas** a `SaveChangesAsync()` y necesitas que si la segunda falla, la primera también se cancele.

Incluso en ese caso, en .NET 10 no necesitas Unit of Work; se maneja directamente con la API de transacciones de EF Core:

```csharp
// Ejemplo en un endpoint de Minimal API (.NET 10) para una sola tabla
app.MapPost("/usuarios/procesar-lote", async (DbContext miContexto) => 
{
    // 1. Abrimos una transacción explícita
    using var transaction = await miContexto.Database.BeginTransactionAsync();

    try 
    {
        // Operación 1
        miContexto.Usuarios.Add(new Usuario { Nombre = "Ana" });
        await miContexto.SaveChangesAsync(); 

        // ... Algo de lógica de negocio intermedia aquí ...

        // Operación 2 (en la misma tabla, pero dependiente de la anterior)
        miContexto.Usuarios.Add(new Usuario { Nombre = "Carlos" });
        await miContexto.SaveChangesAsync();

        // Si todo sale bien, confirmamos
        await transaction.CommitAsync();
        return Results.Ok();
    }
    catch (Exception)
    {
        // Si algo falla, se deshace todo automáticamente al salir del 'using'
        await transaction.RollbackAsync();
        return Results.BadRequest("Error procesando el lote");
    }
});

```

## 🛠️ Resumen de recomendación para .NET 10

* **Para 1 sola tabla:** Inyecta tu `DbContext` directamente en tu Minimal API o Servicio. Haz tus cambios y llama a `SaveChangesAsync()`. Olvídate de Unit of Work y Repositorios.
* **Para proyectos medianos/grandes:** Sigue la misma regla. La documentación oficial de Microsoft y arquitectos de renombre en el ecosistema .NET (como Jimmy Bogard) desaconsejan el uso de Unit of Work sobre EF Core porque bloquea las características avanzadas de LINQ y añade complejidad innecesaria.
