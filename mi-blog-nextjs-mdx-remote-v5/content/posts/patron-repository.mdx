# ¿Qué es el Patrón Repositorio?

El Patrón Repositorio (**Repository Pattern**) es un patrón que **encapsula el acceso a los datos**.

En lugar de que el controlador consulte directamente la base de datos mediante `DbContext`, delega esa responsabilidad a una clase llamada **Repositorio**.

Sin repositorio:

```
Controller
     │
 DbContext
     │
 SQL Server
```

Con repositorio:

```
Controller
     │
 Servicio
     │
Repositorio
     │
 DbContext
     │
 SQL Server
```

La idea es que el controlador **no sepa cómo se obtienen los datos**.

---

# ¿Qué problema resuelve?

Supongamos que tienes un controlador.

```csharp
public class ClienteController : ControllerBase
{
    private readonly FacturacionDbContext _context;

    public ClienteController(FacturacionDbContext context)
    {
        _context = context;
    }

    [HttpGet]
    public async Task<IActionResult> Obtener()
    {
        var clientes = await _context.Clientes.ToListAsync();

        return Ok(clientes);
    }
}
```

Aquí el controlador conoce:

* Entity Framework
* DbContext
* SQL Server

Eso significa que si mañana cambias EF por Dapper, deberás modificar todos los controladores.

# Paso 3. Crear la interfaz del repositorio

Aquí defines **qué puede hacer**.

```csharp
public interface IClienteRepository
{
    Task<List<Cliente>> ObtenerTodosAsync();

    Task<Cliente?> ObtenerPorIdAsync(int id);

    Task AgregarAsync(Cliente cliente);

    Task ActualizarAsync(Cliente cliente);

    Task EliminarAsync(int id);
}
```

Observa que **no aparece Entity Framework**.

Solo reglas de acceso.

---

# Paso 4. Crear la implementación

```csharp
public class ClienteRepository : IClienteRepository
{
    private readonly FacturacionDbContext _context;

    public ClienteRepository(FacturacionDbContext context)
    {
        _context = context;
    }

    public async Task<List<Cliente>> ObtenerTodosAsync()
    {
        return await _context.Clientes.ToListAsync();
    }

    public async Task<Cliente?> ObtenerPorIdAsync(int id)
    {
        return await _context.Clientes.FindAsync(id);
    }

    public async Task AgregarAsync(Cliente cliente)
    {
        _context.Clientes.Add(cliente);

        await _context.SaveChangesAsync();
    }

    public async Task ActualizarAsync(Cliente cliente)
    {
        _context.Clientes.Update(cliente);

        await _context.SaveChangesAsync();
    }

    public async Task EliminarAsync(int id)
    {
        var cliente = await _context.Clientes.FindAsync(id);

        if (cliente == null)
            return;

        _context.Clientes.Remove(cliente);

        await _context.SaveChangesAsync();
    }
}
```

Aquí sí aparece EF Core.

# ¿Es suficiente este repositorio?

Para proyectos pequeños, sí. Sin embargo, en aplicaciones empresariales modernas con **.NET 8, .NET 9 o .NET 10**, es habitual dar un paso más:

* El repositorio **solo se encarga de acceder a los datos**.
* La lógica de negocio (validaciones, reglas, cálculos, etc.) se mueve a una **capa de servicios o casos de uso**.

El flujo queda así:

```
Controller
      │
      ▼
Servicio (Application)
      │
      ▼
Repositorio
      │
      ▼
DbContext
      │
      ▼
SQL Server
```

Por ejemplo, si al crear un cliente debes verificar que el NIT no exista, esa validación pertenece al **servicio**, no al repositorio.

---

## ¿Qué aprender después?

Para dominar el patrón repositorio en proyectos reales, te recomiendo avanzar en este orden:

1. Repositorio básico (CRUD).
2. Inyección de dependencias (`AddScoped`, `AddTransient`, `AddSingleton`).
3. Repositorio genérico (`IRepository<T>`).
4. Unit of Work.
5. Especificaciones (Specification Pattern) para consultas complejas.
6. Clean Architecture integrando Repositorio + Servicios + DTOs + AutoMapper.
7. CQRS con MediatR, donde el repositorio sigue existiendo pero los casos de uso se organizan por comandos y consultas.

## Mi recomendación para ti

Por el tipo de proyectos que has venido desarrollando (facturación, SQL Server, React y APIs REST), **no te quedes en el repositorio básico**. Aprende directamente cómo se implementa en una **Clean Architecture** con:

* DTOs.
* Repositorios.
* Servicios (casos de uso).
* Unit of Work.
* Entity Framework Core.
* Transacciones.

Ese enfoque es el que encontrarás con mayor frecuencia en aplicaciones empresariales de tamaño medio y grande.

Excelente. El siguiente paso es **la capa de Servicios (Service Layer o Application Layer)**, que es donde realmente empieza la separación de responsabilidades. Muchos desarrolladores principiantes cometen el error de poner toda la lógica en el repositorio, cuando en realidad el repositorio **solo debe acceder a los datos**.

# Paso 4. Implementar el servicio

```csharp
public class ClienteService : IClienteService
{
    private readonly IClienteRepository _repository;

    public ClienteService(IClienteRepository repository)
    {
        _repository = repository;
    }

    public async Task CrearClienteAsync(ClienteDto dto)
    {
        // Validar nombre
        if (string.IsNullOrWhiteSpace(dto.Nombre))
            throw new Exception("El nombre es obligatorio.");

        // Validar NIT
        var cliente = await _repository.ObtenerPorNitAsync(dto.Nit);

        if (cliente != null)
            throw new Exception("El NIT ya existe.");

        // Crear entidad
        var nuevo = new Cliente
        {
            Nit = dto.Nit,
            Nombre = dto.Nombre,
            Correo = dto.Correo
        };

        await _repository.AgregarAsync(nuevo);
    }
}
```

Aquí está la diferencia importante.

Toda la lógica está en el servicio.

# Una mejora importante

En los ejemplos anteriores el repositorio ejecuta `SaveChangesAsync()`. En proyectos empresariales esto suele evitarse porque limita el control de las transacciones.

En lugar de eso:

```csharp
public async Task AgregarAsync(Cliente cliente)
{
    await _context.Clientes.AddAsync(cliente);
}
```

Y el guardado se realiza una sola vez desde un **Unit of Work**:

```csharp
await _unitOfWork.SaveChangesAsync();
```

Esto permite que varias operaciones sobre distintos repositorios formen parte de la **misma transacción**, algo muy común en sistemas de facturación.

## El siguiente paso

Ahora que ya entiendes **Controller → Service → Repository**, el siguiente tema natural es **Unit of Work**, donde aprenderás:

* por qué no conviene llamar a `SaveChangesAsync()` dentro de cada repositorio;
* cómo manejar transacciones correctamente;
* cómo coordinar varios repositorios en una sola operación (por ejemplo, crear una factura, sus detalles y actualizar el inventario de forma atómica).

Ese es el patrón que normalmente encontrarás en aplicaciones empresariales desarrolladas con .NET 8, .NET 9 y .NET 10.

Perfecto. Ahora llegamos a uno de los patrones más importantes en aplicaciones empresariales: **Unit of Work (UoW)**.

Este patrón suele ser el punto en el que muchos desarrolladores entienden por qué **no es buena práctica llamar a `SaveChangesAsync()` desde cada repositorio**.

---

# ¿Qué problema resuelve Unit of Work?

Imagina que estás desarrollando un sistema de facturación.

Cuando se crea una factura debes realizar estas operaciones:

1. Guardar la factura.
2. Guardar los detalles.
3. Descontar inventario.
4. Registrar un movimiento de kardex.
5. Registrar una auditoría.

Todo eso debe ocurrir como **una única transacción**.

---

## Sin Unit of Work

Supongamos que cada repositorio hace esto:

```csharp
public async Task AgregarAsync(Factura factura)
{
    _context.Facturas.Add(factura);

    await _context.SaveChangesAsync();
}
```

Luego:

```csharp
public async Task AgregarAsync(FacturaDetalle detalle)
{
    _context.FacturaDetalles.Add(detalle);

    await _context.SaveChangesAsync();
}
```

Y el inventario:

```csharp
public async Task ActualizarAsync(Producto producto)
{
    _context.Productos.Update(producto);

    await _context.SaveChangesAsync();
}
```

Ahora mira el flujo.

```text
Guardar factura

↓

SaveChanges()

↓

Guardar detalle

↓

SaveChanges()

↓

Actualizar inventario

↓

¡¡Error!!
```

¿Qué ocurrió?

La factura quedó guardada.

El detalle quedó guardado.

El inventario NO.

La base de datos quedó inconsistente.

---

# Lo correcto

Debe existir **una sola llamada** a SaveChanges.

```text
Agregar factura

↓

Agregar detalle

↓

Actualizar inventario

↓

Guardar auditoría

↓

SaveChanges()

↓

COMMIT
```

Si algo falla:

```text
ROLLBACK
```

Nada queda guardado.

---

# ¿Quién hace eso?

Unit of Work.

# Paso 2. Crear la interfaz Unit Of Work

```csharp
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync();

    Task BeginTransactionAsync();

    Task CommitAsync();

    Task RollbackAsync();
}
```

---

# Paso 3. Implementación

```csharp
public class UnitOfWork : IUnitOfWork
{
    private readonly FacturacionDbContext _context;

    private IDbContextTransaction? _transaction;

    public UnitOfWork(FacturacionDbContext context)
    {
        _context = context;
    }

    public async Task BeginTransactionAsync()
    {
        _transaction =
            await _context.Database.BeginTransactionAsync();
    }

    public async Task CommitAsync()
    {
        await _context.SaveChangesAsync();

        if (_transaction != null)
            await _transaction.CommitAsync();
    }

    public async Task RollbackAsync()
    {
        if (_transaction != null)
            await _transaction.RollbackAsync();
    }

    public async Task<int> SaveChangesAsync()
    {
        return await _context.SaveChangesAsync();
    }
}
```

# ¿Por qué Microsoft dice que DbContext ya es un Unit of Work?

Porque internamente EF Core ya implementa ese patrón.

De hecho:

```csharp
_context.SaveChangesAsync();
```

ya agrupa todos los cambios pendientes en una sola operación y, por defecto, utiliza una transacción cuando es necesario.

Por eso muchos desarrolladores consideran innecesario crear una clase `UnitOfWork` adicional cuando trabajan **solo con Entity Framework Core**.

---

# Entonces... ¿debo crear Unit of Work?

Depende del proyecto.

## Opción 1 (la que personalmente recomiendo para proyectos modernos con EF Core)

No crear un `UnitOfWork` propio.

```text
Controller

↓

Service

↓

Repository

↓

DbContext
```

El servicio llama una sola vez a:

```csharp
await _context.SaveChangesAsync();
```

Aprovechas el `DbContext` como Unit of Work.

---

## Opción 2 (útil si buscas desacoplar EF Core o mantener una arquitectura muy estricta)

Crear una interfaz `IUnitOfWork` que encapsule `SaveChangesAsync()` y las transacciones.

Esto facilita cambiar la implementación o realizar pruebas con mayor aislamiento.

---

# ¿Cuál es la práctica más utilizada en 2026?

En proyectos nuevos con **.NET 8, .NET 9 y .NET 10**, el enfoque más común es:

* **Repository** para encapsular consultas y persistencia.
* **DbContext** actuando como Unit of Work.
* **SaveChangesAsync() una sola vez por caso de uso**.
* **Transacciones explícitas** solo cuando el caso lo requiere.

Es un enfoque más simple y aprovecha las capacidades nativas de Entity Framework Core.

# ¿Qué es un DTO?

DTO significa **Data Transfer Object**.

Es un objeto que se utiliza **únicamente para transportar información** entre capas o entre el cliente y la API.

No contiene lógica de negocio ni acceso a datos.

---

# ¿Por qué no devolver las entidades?

Supongamos que tu entidad `Cliente` es esta:

```csharp
public class Cliente
{
    public int Id { get; set; }

    public string Nit { get; set; } = "";

    public string Nombre { get; set; } = "";

    public string Correo { get; set; } = "";

    public string Password { get; set; } = "";

    public decimal Saldo { get; set; }

    public DateTime FechaCreacion { get; set; }

    public bool Eliminado { get; set; }
}
```

Si haces esto:

```csharp
return Ok(cliente);
```

El JSON será:

```json
{
  "id": 1,
  "nit": "900123456",
  "nombre": "Empresa ABC",
  "correo": "info@abc.com",
  "password": "123456",
  "saldo": 2500000,
  "fechaCreacion": "2026-07-02",
  "eliminado": false
}
```

**Problemas:**

* Estás enviando la contraseña.
* Envías información que el cliente no necesita.
* Si agregas una columna nueva en la base de datos, también se enviará automáticamente.
* Tu API queda acoplada al modelo de la base de datos.

---

# La solución: DTO

Creamos un DTO específico para responder.

```csharp
public class ClienteResponseDto
{
    public int Id { get; set; }

    public string Nit { get; set; } = "";

    public string Nombre { get; set; } = "";

    public string Correo { get; set; } = "";
}
```

Ahora el JSON será:

```json
{
   "id":1,
   "nit":"900123456",
   "nombre":"Empresa ABC",
   "correo":"info@abc.com"
}
```

Mucho más limpio.

# ¿AutoMapper sigue siendo la mejor opción en .NET 8, 9 y 10?

Aquí hay un cambio importante en el ecosistema.

Hace algunos años, **AutoMapper** era prácticamente el estándar. Sin embargo, hoy muchas empresas prefieren realizar el mapeo **manual** o utilizar **generadores de código** como **Mapperly** o **Mapster**, porque ofrecen mejor rendimiento y permiten detectar errores en tiempo de compilación.

En proyectos pequeños y medianos, el mapeo manual suele ser suficiente y muy fácil de mantener.

Por ejemplo:

```csharp
public static class ClienteMapper
{
    public static ClienteResponseDto ToResponseDto(this Cliente cliente)
    {
        return new ClienteResponseDto
        {
            Id = cliente.Id,
            Nombre = cliente.Nombre,
            Correo = cliente.Correo
        };
    }

    public static Cliente ToEntity(this ClienteCreateDto dto)
    {
        return new Cliente
        {
            Nit = dto.Nit,
            Nombre = dto.Nombre,
            Correo = dto.Correo,
            Password = dto.Password
        };
    }
}
```

Uso:

```csharp
var dto = cliente.ToResponseDto();
```

Este enfoque tiene varias ventajas:

* El código es explícito y fácil de seguir.
* El compilador detecta cambios en propiedades.
* No dependes de reflexión en tiempo de ejecución.
* Es sencillo depurarlo.

## El siguiente paso será **Validaciones profesionales con FluentValidation**.

Aprenderás a eliminar validaciones del tipo:

```csharp
if (string.IsNullOrWhiteSpace(dto.Nombre))
    throw new Exception("El nombre es obligatorio.");
```

y reemplazarlas por validadores reutilizables, desacoplados y fáciles de mantener. Este es el enfoque que encontrarás con frecuencia en proyectos modernos desarrollados con ASP.NET Core 8, 9 y 10.
