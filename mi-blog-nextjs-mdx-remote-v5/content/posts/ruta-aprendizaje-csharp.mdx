## Donde deben ir las validaciones:

Es una muy buena pregunta. En **2026**, la respuesta depende de la arquitectura que estés utilizando. Para una **API REST en .NET con arquitectura en capas o Clean Architecture**, **no se recomienda poner todas las validaciones en las entidades lanzando excepciones**.

Lo ideal es **separar los tipos de validaciones**.

---

# 1. Validaciones de reglas de negocio → En las entidades (sí)

Las entidades deben proteger su estado y garantizar que nunca queden inválidas.

Por ejemplo:

```csharp
public class Product
{
    public string Name { get; private set; }

    public Product(string name)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new DomainException("El nombre es obligatorio.");

        Name = name;
    }

    public void ChangePrice(decimal price)
    {
        if (price <= 0)
            throw new DomainException("El precio debe ser mayor que cero.");

        Price = price;
    }
}
```

Estas son **reglas del dominio**.

---

# 2. Validaciones de entrada → No en la entidad

Si llega un DTO desde la API:

```json
{
  "name": "",
  "price": -5
}
```

No conviene que la entidad sea la primera que detecte estos errores.

Es mejor validar el DTO antes.

Por ejemplo con **FluentValidation**:

```csharp
public class CreateProductValidator : AbstractValidator<CreateProductDto>
{
    public CreateProductValidator()
    {
        RuleFor(x => x.Name)
            .NotEmpty()
            .MaximumLength(100);

        RuleFor(x => x.Price)
            .GreaterThan(0);
    }
}
```

Así la API responde con un **400 Bad Request** antes de llegar al dominio.

---

# 3. Validaciones de infraestructura

Ejemplos:

- ¿Existe el cliente?
- ¿Existe el producto?
- ¿El correo ya está registrado?
- ¿El NIT ya existe?

Estas requieren consultar la base de datos.

No pertenecen a la entidad.

Normalmente van en:

- Application Service
- Command Handler
- Domain Service (si involucran lógica de dominio)
- Repositorios (solo para consultar datos)

---

# ¿Entonces las entidades nunca lanzan excepciones?

Sí pueden lanzarlas.

Pero normalmente son **excepciones del dominio**, por ejemplo:

```csharp
throw new DomainException("No puede crear una factura sin detalles.");
```

No deberían lanzar:

```csharp
throw new Exception();
```

ni

```csharp
throw new SqlException(...);
```

```

```
DomainException
```

```

# ¿Y las DataAnnotations?

Como:

```csharp
[Required]
[StringLength(50)]
```

Son útiles para proyectos pequeños.

En proyectos grandes hoy suele preferirse **FluentValidation** porque:

- separa las validaciones
- es más flexible
- permite validaciones complejas
- es fácil de probar

---

# Lo que yo haría en un proyecto moderno (.NET 8 o superior)

```
Controllers
        │
        ▼
FluentValidation
        │
        ▼
Application Service
        │
        ▼
Entidades (reglas de negocio)
        │
        ▼
Repositorio
        │
        ▼
SQL Server
```

# Para tu proyecto de facturación

Como sé que estás desarrollando una API con:

- .NET 8
- SQL Server
- JWT
- Dapper o EF Core
- arquitectura en capas

Yo usaría:

- ✅ **FluentValidation** para validar los DTOs que llegan desde la API.
- ✅ **Entidades** para proteger las reglas de negocio e impedir estados inválidos.
- ✅ **Excepciones de dominio** (`DomainException`) solo cuando se viola una regla del negocio.
- ✅ **Middleware de manejo de excepciones** para transformar esas excepciones en respuestas HTTP adecuadas (400, 404, 409, etc.).

Ese enfoque mantiene cada responsabilidad en su lugar, hace el código más fácil de mantener y de probar, y es el que más suele verse en aplicaciones .NET empresariales modernas.
