**Nota** los nombres de las tablas en postgresql deben ser en minuscula y separadao con **\_** guion abajo
🛠️ El secreto para conectarlo con .NET 10 (Sin ensuciar tu código)
Si usas la estructura anterior (snake_case), no tienes por qué castigar tu código de C# escribiendo propiedades en minúsculas.

El proveedor oficial de Postgres para Entity Framework Core (Npgsql) incluye una solución elegante. Solo debes configurar la convención de nombres en tu DbContext para que haga la traducción automática de PascalCase (C#) a snake_case (Postgres).

1. Instala el paquete NuGet de extensiones de nombres:

```bash
dotnet add package EFCore.NamingConventions
```

2. Configura tu DbContext en .NET 10:
   Al registrar tu contexto (ya sea en el Program.cs o en el OnConfiguring), añades .UseSnakeCaseNamingConvention():

```csharp
protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
{
    optionsBuilder
        .UseNpgsql("TuCadenaDeConexion")
        .UseSnakeCaseNamingConvention(); // <--- ¡Aquí está la magia!
}
```

3. Tu Entidad en C# queda perfecta y limpia:
   Gracias a esa línea, EF Core sabe que cuando en C# preguntas por PrimerNombre, en Postgres debe buscar primer_nombre.

```csharp
public class Tercero
{
    public long Id { get; set; }
    public string Nit { get; set; } = null!;
    public string PrimerNombre { get; set; } = null!; // Se mapea automáticamente a primer_nombre
    public string? SegundoNombre { get; set; }
    public string RazonSocial { get; set; } = "";
    // ...
}
```

**CONCLUSION**
manejare sqlserver con las convencion de postgresql y el codigo en csharp quedaria:

```csharp
protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
{
    // Supongamos que lees el tipo de motor desde tu appsettings.json
    string motor = Configuration["DatabaseProvider"];

    if (motor == "PostgreSQL")
    {
        optionsBuilder.UseNpgsql(Configuration.GetConnectionString("PostgresConnection"));
    }
    else if (motor == "SqlServer")
    {
        optionsBuilder.UseSqlServer(Configuration.GetConnectionString("SqlServerConnection"));
    }

    // ¡La magia universal! Aplica snake_case automáticamente para CUALQUIER motor
    optionsBuilder.UseSnakeCaseNamingConvention();
}
```

## Carga masiva desde netcore

Opción 3: Carga masiva desde .NET 10 (C#)
Como estás usando .NET 10, puedes programar la carga masiva de manera ultra rápida usando Npgsql (el proveedor de PostgreSQL para .NET).  
Esto evita tocar permisos del sistema de archivos del servidor y usa la infraestructura de streaming de Postgres:

```sql
using var dataSource = NpgsqlDataSource.Create("tu_cadena_conexion");
await using var connection = await dataSource.OpenConnectionAsync();

// Abre el archivo CSV desde C#
using var reader = new StreamReader(@"C:\tribusoft\excel\dptos.csv");

// Inicia la operación de copia masiva binaria o de texto
await using var writer = await connection.BeginTextImportAsync(
    "COPY departamento (codigo, nombre) FROM STDIN WITH DELIMITER ';' CSV HEADER"
);

// Transmite el contenido del archivo directamente a la base de datos
string? line;
while ((line = await reader.ReadLineAsync()) != null)
{
    await writer.WriteLineAsync(line);
}
```

Con este enfoque de .NET, tu usuario de base de datos solo necesita permisos estándar de INSERT sobre la tabla departamento.  
No requiere permisos de servidor.


No usar **public partial class** sino simplemente **public class**  
En .NET 10 (2026) y siguiendo las recomendaciones de Microsoft y las prácticas utilizadas en proyectos empresariales, la regla es bastante sencilla:

Usa public class en prácticamente todas las clases que escribas manualmente.
Usa public partial class únicamente cuando la clase se divide intencionalmente en varios archivos o una parte es generada automáticamente por una herramienta.

No usar **virtual** este codigo se usa para carga peresosa ya no se recomienda

**Principales mejoras aplicadas**
Convenciones de C# 10/11: **record** para entidades inmutables o class **sealed** si necesitas herencia controlada.

Data Annotations: validación y restricciones directamente en el modelo.

Propiedades requeridas: uso de **required** en lugar de inicialización con **null!**.

Nombres consistentes: claridad en nombres de propiedades.

Relaciones EF Core: navegación explícita y clave foránea.

Separación de capas: entidad de dominio sin lógica de persistencia.

```sql
namespace TribuSoft.Domain.Entities;

public sealed class Departamento
{
    // Usamos 'init' ya que el código suele ser la llave primaria natural y no cambia tras crearse.
    public string Codigo { get; private init; }
    public string Nombre { get; private set; }

    // Respaldamos la lista en un campo privado
    private readonly List<Municipio> _municipios = [];

    // Exponemos la colección como de solo lectura para proteger la consistencia del dominio
    public IReadOnlyCollection<Municipio> Municipios => _municipios;

    // Constructor requerido por Entity Framework Core para la materialización
    private Departamento() { }

    // Constructor recomendado para crear la entidad de forma controlada
    public Departamento(string codigo, string nombre)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(codigo);
        ArgumentException.ThrowIfNullOrWhiteSpace(nombre);

        Codigo = codigo;
        Nombre = nombre;
    }

    // Método de negocio para encapsular cómo se añade un municipio
    public void RegistrarMunicipio(Municipio municipio)
    {
        ArgumentNullException.ThrowIfNull(municipio);

        // Aquí podrías agregar validaciones de negocio antes de insertar
        _municipios.Add(municipio);
    }
}
```

y la configuracion seria:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using TribuSoft.Domain.Entities;

namespace TribuSoft.Infrastructure.Persistence.Configurations;

public sealed class DepartamentoConfiguration : IEntityTypeConfiguration<Departamento>
{
    public void Configure(EntityTypeBuilder<Departamento> builder)
    {
        builder.ToTable("Departamentos");

        // Configuración de la Llave Primaria
        builder.HasKey(d => d.Codigo);

        builder.Property(d => d.Codigo)
            .HasMaxLength(5) // Ajusta al tamaño de tus códigos (ej. ISO o DIVIPOLA)
            .IsRequired();

        builder.Property(d => d.Nombre)
            .HasMaxLength(100)
            .IsRequired();

        // Configuración de la relación 1 a Muchos con acceso al campo privado (Backing Field)
        builder.HasMany(d => d.Municipios)
            .WithOne(m => m.Departamento) // Asumiendo que Municipio tiene la propiedad 'Departamento'
            .HasForeignKey(m => m.CodigoDepartamento)
            .Metadata.PrincipalToDependent?
            .SetPropertyAccessMode(PropertyAccessMode.Field);
    }
}
```

**Mejorias aplicadas**

```sql
¿Qué mejoras y convenciones se aplicaron?
Movimiento al Dominio (Domain.Entities):
La entidad ahora vive en el corazón de la aplicación. No depende de bases de datos, ORMs ni librerías externas.

Uso de expresiones de colección [] (C# 12+):
Se reemplazó el tradicional new List<Municipio>() por la sintaxis moderna y eficiente de inicialización de colecciones [].

Encapsulamiento de Colecciones (IReadOnlyCollection):
Si dejas public ICollection<Municipio> Municipios { get; set; },
cualquier parte de tu código (como un controlador o servicio) podría hacer departamento.Municipios.Clear()
o reasignar la lista directamente, rompiendo las reglas del negocio.
Exponerla como IReadOnlyCollection obliga a usar métodos controlados (como RegistrarMunicipio).

ArgumentException.ThrowIfNullOrWhiteSpace (C# 12/13):
Validaciones nativas y optimizadas de rendimiento en los constructores para asegurar
que un Departamento nunca nazca con datos corruptos o vacíos.

Configuración segregada:
La infraestructura ahora se limita puramente a mapear la entidad mediante IEntityTypeConfiguration<T>,
logrando un desacoplamiento total y un código sumamente limpio.

```

```sql
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

// Lee el proveedor desde appsettings.json
var provider = builder.Configuration["DatabaseProvider"];
var connectionString = builder.Configuration.GetConnectionString(provider);

builder.Services.AddDbContext<AppDbContext>(options =>

    }

var app = builder.Build();

app.Run();

```

# 12. Tips que se deben tener en cuenta al momento de crear las entidades:

Si hablamos de **.NET 10 en 2026**, hay varias prácticas que hace unos años eran muy populares pero que hoy han perdido protagonismo o ya no se consideran la mejor opción. Te resumo las recomendaciones actuales centrándome en **entidades de dominio (DDD + Clean Architecture)**.

---

# 12.1. La entidad representa el negocio, no la base de datos ⭐⭐⭐⭐⭐

Este es el principio más importante.

Una entidad debe modelar el dominio, no la tabla de la base de datos.

**Correcto**

```csharp
public sealed class Departamento
{
    public required string Codigo { get; init; }

    public required string Nombre { get; private set; }

    public void CambiarNombre(string nombre)
    {
        Nombre = nombre;
    }
}
```

No pienses:

> "Tengo una tabla Departamento"

Piensa:

> "Tengo un Departamento del negocio."

---

# 12.2. No usar DataAnnotations en las entidades ⭐⭐⭐⭐⭐

En 2026 prácticamente todos los proyectos nuevos usan **Fluent API**.

Evita:

```csharp
[Table("Departamento")]
[Key]
[Required]
[StringLength(100)]
```

Todo eso pertenece a Infrastructure.

---

# 12.3. Configurar EF Core mediante IEntityTypeConfiguration ⭐⭐⭐⭐⭐

```text
Infrastructure
    Persistence
        Configurations
            DepartamentoConfiguration.cs
```

```csharp
public sealed class DepartamentoConfiguration
    : IEntityTypeConfiguration<Departamento>
{
    public void Configure(EntityTypeBuilder<Departamento> builder)
    {
        builder.HasKey(x => x.Codigo);

        builder.Property(x => x.Nombre)
               .HasMaxLength(100);
    }
}
```

Esta es la forma recomendada actualmente.

---

# 12.4. Usar `required` cuando el dato es obligatorio ⭐⭐⭐⭐⭐

Desde C# 11 y .NET 10 es completamente válido.

```csharp
public required string Codigo { get; init; }

public required string Nombre { get; private set; }
```

No hace falta escribir constructores solo para evitar `null`.

---

# 12.5. Preferir `init` para propiedades inmutables ⭐⭐⭐⭐⭐

Si el dato nunca cambia:

```csharp
public required string Codigo { get; init; }
```

Ejemplos:

* Código
* Número de documento
* Fecha de creación (si no cambia)

---

# 12.6. `private set` para propiedades modificables ⭐⭐⭐⭐⭐

```csharp
public required string Nombre { get; private set; }

public void CambiarNombre(string nombre)
{
    Nombre = nombre;
}
```

No permitir:

```csharp
departamento.Nombre = "...";
```

---

# 12.7. No exponer `List<T>` ⭐⭐⭐⭐⭐

En lugar de:

```csharp
public List<Ciudad> Ciudades { get; set; }
```

usar:

```csharp
private readonly List<Ciudad> _ciudades = [];

public IReadOnlyCollection<Ciudad> Ciudades
    => _ciudades;
```

---

# 12.8. Encapsular el comportamiento ⭐⭐⭐⭐⭐

Evita entidades "anémicas" con solo propiedades.

Mal:

```csharp
public string Nombre { get; set; }
```

Mejor:

```csharp
public void CambiarNombre(string nombre)
{
    // Validaciones
    Nombre = nombre;
}
```

---

# 12.9. Validar invariantes en la entidad ⭐⭐⭐⭐⭐

```csharp
public void CambiarNombre(string nombre)
{
    ArgumentException.ThrowIfNullOrWhiteSpace(nombre);

    if (nombre.Length > 100)
        throw new DomainException("Nombre inválido.");

    Nombre = nombre;
}
```

La entidad protege sus propias reglas de negocio.

---

# 12.10. No crear DTOs dentro del dominio ⭐⭐⭐⭐⭐

Incorrecto:

```text
Domain
    DTOs
```

Los DTO pertenecen a Application o a la API.

---

# 12.11. Evitar repositorios genéricos ⭐⭐⭐⭐⭐

Hace algunos años era muy común:

```csharp
IGenericRepository<T>
```

Hoy suele considerarse una abstracción demasiado genérica para dominios ricos. Es preferible definir repositorios orientados al negocio cuando realmente se necesitan.

Ejemplo:

```csharp
IDepartamentoRepository
```

---

# 12.12. No agregar constructores innecesarios ⭐⭐⭐⭐⭐

Hace años era habitual:

```csharp
private Departamento()
{
}
```

Hoy EF Core puede trabajar perfectamente con constructores parametrizados y propiedades `init` o con un constructor sin parámetros cuando sea necesario. No agregues un constructor vacío solo "porque EF lo necesita"; añádelo únicamente si realmente tu diseño lo requiere.

---

# 12.13. Evitar clases `partial` ⭐⭐⭐⭐⭐

No uses:

```csharp
public partial class Departamento
```

Salvo cuando:

* el código es generado automáticamente;
* trabajas con WinForms, Razor u otra herramienta que lo requiera.

Para entidades de dominio:

```csharp
public sealed class Departamento
```

es lo habitual.

---

# 12.14. Preferir `sealed` ⭐⭐⭐⭐⭐

```csharp
public sealed class Departamento
```

Si nadie va a heredar de la entidad, declara la clase `sealed`. Esto expresa claramente la intención del diseño.

---

# 12.15. No incluir lógica de acceso a datos ⭐⭐⭐⭐⭐

Incorrecto:

```csharp
public async Task Guardar()
```

o

```csharp
public async Task Eliminar()
```

La entidad no conoce EF Core, SQL ni Dapper.

---

# 12.16. Usar Value Objects cuando corresponda ⭐⭐⭐⭐⭐

En lugar de:

```csharp
public string Correo { get; set; }
```

preferir:

```csharp
public Email Email { get; private set; }
```

cuando el concepto tenga reglas propias.

---

# 12.17. No usar Lazy Loading ⭐⭐⭐⭐⭐

Actualmente se suele evitar porque:

* produce consultas inesperadas (N+1);
* dificulta medir el rendimiento;
* hace menos evidente cuándo se accede a la base de datos.

Se prefieren consultas explícitas (`Include`, proyecciones o CQRS).

---

# 12.18. Separar claramente Domain de Infrastructure ⭐⭐⭐⭐⭐

El proyecto Domain no debería referenciar:

* EF Core
* Dapper
* Npgsql
* SQL Server
* AutoMapper

Debe poder compilar sin ninguna dependencia de infraestructura.

---

# 12.19. Mantener las entidades pequeñas ⭐⭐⭐⭐⭐

Una entidad con cientos de líneas suele indicar que conviene extraer Value Objects o servicios de dominio.

---

# 12.20. Evitar código generado dentro del dominio ⭐⭐⭐⭐⭐

Si utilizas `Scaffold-DbContext`, no coloques las clases generadas dentro de `Domain`. Lo habitual es mantenerlas en `Infrastructure/Persistence` o usarlas solo como apoyo para el mapeo, dejando el dominio libre de código generado.

---

# Lo que hoy se considera desactualizado o menos recomendable

```

| Práctica                                    | Estado en 2026 | Alternativa recomendada                       |
| ------------------------------------------- | -------------- | --------------------------------------------- |
| DataAnnotations para mapear EF Core         | ❌             | Fluent API                                    |
| Repositorio genérico para todo              | ⚠️             | Repositorios específicos cuando aporten valor |
| Entidades anémicas (solo propiedades)       | ❌             | Entidades con comportamiento                  |
| `List<T>` públicas                          | ❌             | `IReadOnlyCollection<T>` con respaldo privado |
| `public set` en todas las propiedades       | ❌             | `private set` o `init`                        |
| Constructores vacíos añadidos por costumbre | ⚠️             | Solo cuando el diseño lo requiera             |
| Lógica SQL dentro de entidades              | ❌             | Infrastructure                                |
| DTOs en Domain                              | ❌             | Application/API                               |
| Lazy Loading como estrategia por defecto    | ❌             | Carga explícita                               |
```

## Para TribuSoft

Por las preguntas que hemos venido trabajando, la arquitectura que considero más alineada con las prácticas actuales sería:

* **Domain**: entidades ricas, `sealed`, `required`, `init` y `private set`, comportamiento encapsulado y Value Objects cuando aporten valor.
* **Application**: casos de uso, DTOs, validaciones de entrada y orquestación.
* **Infrastructure**: EF Core 10, `DbContext`, configuraciones con Fluent API, repositorios y migraciones.
* **API**: endpoints, autenticación, serialización y composición de dependencias.

Esa combinación refleja el enfoque que hoy es más habitual en proyectos profesionales modernos con **.NET 10**, evitando patrones que han quedado obsoletos o que ya no aportan beneficios claros.
