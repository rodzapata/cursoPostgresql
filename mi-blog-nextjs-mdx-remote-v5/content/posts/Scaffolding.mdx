# 2. Estructura de Aquitectura Limpia

La **Arquitectura Limpia (Clean Architecture)**, propuesta por **Robert C. Martin (Uncle Bob)**, organiza una aplicación en capas con una regla fundamental:

> **Las dependencias siempre apuntan hacia el centro.** Las capas internas no conocen las externas.

En un proyecto de **.NET 8**, una implementación típica tiene entre **4 y 5 capas**.

```text
┌──────────────────────────────────────────┐
│              Presentation                │
│     API, MVC, Blazor, Minimal API        │
└──────────────────────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────┐
│             Application                  │
│ Casos de uso, DTO, Interfaces, Servicios │
└──────────────────────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────┐
│                Domain                    │
│ Entidades, Reglas de negocio, ValueObjects│
└──────────────────────────────────────────┘
                    ▲
                    │
┌──────────────────────────────────────────┐
│            Infrastructure                │
│ EF Core, SQL Server, PostgreSQL, JWT     │
│ Email, Archivos, APIs externas           │
└──────────────────────────────────────────┘
```

---

# 1. Domain (Núcleo)

Es la capa más importante.

Aquí vive únicamente el negocio.

No debe conocer:

- Entity Framework
- SQL Server
- ASP.NET
- JWT
- HTTP
- JSON

Contiene:

- Entidades
- Value Objects
- Enums
- Excepciones de dominio
- Interfaces de dominio (cuando aplica)
- Reglas de negocio

Ejemplo:

```csharp
public class Producto
{
    public Guid Id { get; private set; }

    public string Nombre { get; private set; }

    public decimal Precio { get; private set; }

    public void CambiarPrecio(decimal nuevoPrecio)
    {
        if (nuevoPrecio <= 0)
            throw new Exception("Precio inválido");

        Precio = nuevoPrecio;
    }
}
```

Observa que aquí no existe:

- DbContext
- SQL
- Controllers
- JSON

---

# 2. Application

Aquí viven los **casos de uso**.

Es donde se implementa la lógica de la aplicación utilizando las entidades del dominio.

Contiene normalmente:

- DTO
- Interfaces de repositorios
- Interfaces de servicios
- Casos de uso
- Commands
- Queries
- Validaciones
- AutoMapper

Ejemplo:

```text
Application

DTO
    ProductoDto

Interfaces
    IProductoRepository

Services
    ProductoService

Commands
Queries
Validators
```

Ejemplo:

```csharp
public class ProductoService
{
    private readonly IProductoRepository _repository;

    public ProductoService(IProductoRepository repository)
    {
        _repository = repository;
    }

    public async Task<List<ProductoDto>> ObtenerTodos()
    {
        var productos = await _repository.ObtenerTodos();

        return productos.Select(x => new ProductoDto
        {
            Id = x.Id,
            Nombre = x.Nombre
        }).ToList();
    }
}
```

Aquí solo se conoce la interfaz.

No se conoce SQL Server.

---

# 3. Infrastructure

Aquí están las implementaciones.

Todo lo relacionado con tecnología.

Ejemplos:

- Entity Framework Core
- SQL Server
- PostgreSQL
- Oracle
- MongoDB
- JWT
- Email
- Azure
- AWS
- Redis
- RabbitMQ

Ejemplo:

```text
Infrastructure

Persistence
    AppDbContext

Repositories
    ProductoRepository

Security
    JwtService

Email
    EmailService
```

Ejemplo:

```csharp
public class ProductoRepository : IProductoRepository
{
    private readonly AppDbContext _context;

    public ProductoRepository(AppDbContext context)
    {
        _context = context;
    }

    public async Task<List<Producto>> ObtenerTodos()
    {
        return await _context.Productos.ToListAsync();
    }
}
```

La aplicación conoce la interfaz.

Infrastructure implementa esa interfaz.

---

# 4. Presentation

Es la entrada al sistema.

Puede ser:

- Web API
- MVC
- Blazor
- MAUI
- gRPC

Aquí viven:

- Controllers
- Endpoints
- Middleware
- Swagger
- Filtros
- Configuración

Ejemplo:

```text
Presentation

Controllers

ProductosController

Program.cs

appsettings.json
```

Controller:

```csharp
[ApiController]
[Route("api/productos")]
public class ProductosController : ControllerBase
{
    private readonly ProductoService _service;

    public ProductosController(ProductoService service)
    {
        _service = service;
    }

    [HttpGet]
    public async Task<IActionResult> Get()
    {
        return Ok(await _service.ObtenerTodos());
    }
}
```

El controlador no consulta la base de datos directamente.

# Dependencias

La regla de Clean Architecture es:

```text
Presentation
      │
      ▼
Application
      │
      ▼
Domain

Infrastructure
      │
      └────────► Domain
      └────────► Application
```

Es decir:

- **Domain** no depende de nadie.
- **Application** depende de **Domain**.
- **Infrastructure** depende de **Application** y **Domain**.
- **Presentation** depende de **Application** (y normalmente registra las implementaciones de Infrastructure mediante inyección de dependencias).

## ¿Es esta la arquitectura más recomendada para .NET 8?

Sí. Para aplicaciones empresariales (APIs REST, sistemas de facturación, ERP, CRM, etc.), esta estructura es una de las más utilizadas porque:

- Mantiene separadas las responsabilidades.
- Facilita las pruebas unitarias al depender de interfaces.
- Permite cambiar tecnologías (por ejemplo, SQL Server por PostgreSQL) con un impacto mínimo en la lógica de negocio.
- Escala mejor a medida que crece el proyecto.

Dado que en conversaciones anteriores has comentado que estás desarrollando APIs REST en **.NET 8** con **Entity Framework Core**, **JWT**, **DTO** y **SQL Server/PostgreSQL**, esta arquitectura encaja muy bien con ese tipo de proyectos y es una excelente base para construir aplicaciones mantenibles y fáciles de evolucionar.
